Motivation

We have already seen that a simple linear model estimates the population relationship between a response (Y) and a predictor (X) using a sample. The relationship being estimated is \[Y = \beta_0 + \beta_1 X + \epsilon\] where we use the process of least squares to estimate a slope \(\beta_1\) and an intercept \(\beta_0\).

We have also now seen that we can use multiple predictors in a model to estimate a more complex relationship of the form \[Y = \beta_0 + \beta_1 X_1 + \beta_2 X_2 + \ldots + \beta_p X_p + \epsilon.\] where the \(p\) represents the number of predictors used in the model. While it is common that our variables are continuous in nature, there are many characteristics/properties that are qualitative in nature that are useful in understanding the population relationship. When working with qualitative predictors, we need to treat them a little bit differently than continuous ones.

In this module, we will explore how we can use qualitative predictors in a multiple linear model to estimate population relationships, and how this changes our interpretation of the coefficients in the model.

Section 1: Working with Categorical/Qualitative Predictors

Qualitative variables are really useful as predictors, but can be difficult to work with. Before seeing how we can use them in a model, we will look at some skills for manipulating the different types of qualitative variables.

Named Categories

Qualitative predictors are most recognizable when the values they take are names or words, such as the variable Theme in our LEGO dataset. They can sometimes be tricky to work with, so it’s important to know how they differ from quantitative variables.

The first important step in working with qualitative variables is to know how to subset or select only certain levels/values of the variable. To do this, we can use true/false conditions (called booleans) to check is a certain row holds a specific value or not. We use variable == "value" to check this. We can also check whether a certain row does not hold a specific value by using variable != "value".

The next important skill is to know how to convert the text values of the levels of a qualitative variable into indicator variables. This is useful for understanding how R treats qualitative information when used in a linear model. The Theme variable has 3 possible values: City, Friends, DUPLO\xae. Lastly, we can see how we can use this type of variable in a simple linear model. When the levels are words/text, we can simply write the variable name as a predictor and the model will understand to treat it as qualitative.

# create a simple linear model with Theme as a predictor
lm(Amazon_Price ~ ___, data=lego)
lm(Amazon_Price ~ as.factor(Theme), data=lego)

Hint: Remember that the predictor we want to include is Theme.

What we see here is that the coefficients are labelled with the value of the qualitative predictor, and that we only see two out of the three levels present. This is because the intercept is actually equal to the average value of Amazon_Price among the City sets: E(Y | X = “City”) = 45.2696. The slope for Friends is the difference between the mean Amazon Price of City sets and Friends sets: E(Y | X = “Friends”) - E(Y | X = “City”) = -6.6204.

We could have also fit the exact same model using indicator variables (0’s and 1’s), we would just need to include more than one predictor.

Numerical Categories

Sometimes, qualitative information is recorded as numbers, usually when we are dealing with a rating system. If we were to use this kind of variable as a predictor, R would treat this as a numerical variable. Let’s fit a simple linear model to Amazon_Price using rating as a predictor to see what we get.

# fit a model using rating as a quantitative predictor
lm(Amazon_Price ~ ___, data=lego)
lm(Amazon_Price ~ rating, data=lego)

Since R treats the predictor as continuous, we can interpret the coefficient as we normally would. However, we should re-evaluate whether it makes sense to treat the predictor as continuous. The rating scale only goes from 1 to 5, and it cannot take values that are not integers. Further, it is inherently a qualitative measurement (someone judged the set on goodness and gave it a rating). It also may not make sense to assume that, if we compare sets with a rating of 1 and 2, it would necessarily have the same effect on the mean Amazon Price as sets with ratings of 4 and 5 (i.e. the change may not be the same for different ratings).

Therefore, it’s important to always think about the variables to best decide how to use theme in a model. We can tell R that we want it to use rating as a qualitative variable by using the as.factor(variable) command, which forces R to see this variable as qualitative rather than numerical. Try this below to see how it changes the model estimates.

# fit a model by including rating as a qualitative variable
lm(Amazon_Price ~ ___(rating), data=lego)
lm(Amazon_Price ~ as.factor(rating), data=lego)
# to make rating a qualitative variable, use
as.factor(rating)

Notice that the rating of 1 is not present in the model, as the intercept captures that information. Again we would interpret the coefficients as the change in the average Amazon Price of a set with rating of X compared to a rating of 1.

The results of a model that uses ratings as a categorical predictor shows us that, compared to a rating of 1, sets with other ratings change the mean Amazon Price by different amounts, so it may not be reasonable to treat this predictor as continuous with a constant effect on the response.

Section 3: Indicators in Interaction Terms

We just learned that using an indicator/qualitative variable as a main effect predictor results in an estimated relationship for City that is parallel to the estimated relationship for Friends. In other words, using an indicator variable as a main effect gives us two lines with the same slope but different intercepts.

Let’s consider a different scenario, one in which we have two predictors that work together or interact with each other to define the relationship between Amazon_Price and Pieces. We say that two variables interact when the effect of one variable on the response depends on the value of the other variable.

To see an example of this, we will be working with the Theme variable again, but this time investigating the DUPLO sets.

Section 4: Indicators in both Interactions and Main Effects

As we have now seen, it’s possible to use qualitative variables to denote groups of like measurements, and specify different linear relationships for each of these groups through a multiple linear model. We have seen that using qualitative information as a main effect yields relationships with different intercepts, while using it in an interaction term yields relationships with different slopes.

You might be wondering whether it’s possible to combine these to specify a relationship with different slopes and intercepts for each group. The answer of course is YES! We can explore how to do this below.

Estimating Different Slopes and Intercepts for Three Groups

From what we have seen before, we know from Section 2 that City sets and Friends sets have different intercepts but similar slopes. And from Section 3, we know that City sets and DUPLO sets have common intercepts but different slopes. Let’s compare simple linear models for Friends sets and DUPLO sets to understand whether there are differences in their relationship.

# fit the simple linear models for Friends and DUPLO
slr_friends <- lm(Amazon_Price ~ Pieces, data = ____)
slr_friends

slr_duplo <- lm(Amazon_Price ~ Pieces, data = ____)
slr_duplo
slr_friends <- lm(Amazon_Price ~ Pieces, data = lego[lego$Theme == "Friends", ])
slr_friends

slr_duplo <- lm(Amazon_Price ~ Pieces, data = lego[lego$Theme == "DUPLO\xae", ])
slr_duplo
# remember that to isolate the data from only one group, we use
lego[lego$Theme == "DUPLO\xae", ]

What you probably see from the results is that the relationship between Amazon Price and Pieces is different depending on the Theme (they have both different slopes and intercepts). To fit this relationship using a single model, we would have to include both a main effect AND an interaction term to create this difference. We would do this by specifying a model with the form lm(response ~ predictor1 + predictor2 + predictor1:predictor2), or perhaps more simply lm(response ~ predictor1*predictor2).

Try both versions below, making sure to use a dataset that excludes the City sets.

# fit a model by explicitly entering all three terms
lm(Amazon_Price ~ ___ + ___ + ___, data = lego[lego$Theme != "City",])

# fit the same model but using the simpler syntax
lm(Amazon_Price ~ _____, data = ____)
lm(Amazon_Price ~ Pieces + Theme + Pieces:Theme, data = lego[lego$Theme != "City",])


lm(Amazon_Price ~ Pieces:Theme, data = lego[lego$Theme != "City",])
# the more explicit way is
lm(response ~  predictor1 + predictor2 + predictor1:predictor2)
# while the simpler way is
lm(response ~ predictor1*predictor2)
# where predictor1 = Pieces and predictor2 = Theme

Customizing the Relationship

If you look closely at the model estimates, you will notice that the slope for the interaction of Pieces and Friends Theme is very small and close to 0. We also noticed through our explorations that the Friends and City Themes had almost the same slope in their respective simple linear models. So you might wonder whether it’s necessary to estimate a value for this slope at all.

It’s definitely possible (as we’ll see below), but most of the time, it’s easier to leave the qualitative variables as they are, rather than change them to individual indicator variables. But since we’ve learned how to do this in Section 1, we can apply the same technique here.

First we will separate the qualitative variable into indicators. Then we can fit a multiple linear model that will only interact Pieces with the DUPLO set and not with the Friends set.

# create indicator variables for the levels of the correct answer above
ind1 <- ifelse(____, 1, 0)
ind2 <- ifelse(____, 1, 0)

# fit the model described above
lm(Amazon_Price ~ Pieces + ___ + ___ + Pieces:___, data = lego)
ind1 <- ifelse(lego$Theme == "City", 1, 0)
ind2 <- ifelse(lego$Theme == "DUPLO\xae", 1, 0)


lm(Amazon_Price ~ Pieces + ind1 + ind2 + Pieces:ind2, data = lego)
# remember that the condition we wish to check is, for example
lego$Theme == "DUPLO\xae"
# remember that the condition we wish to check is, for example
lego$Theme == "DUPLO\xae"

# We want the first two gaps to be filled by `ind1` and `ind2` and the third gap to be filled by whichever one of `ind1` and `ind2` holds the information for the DUPLO set.

In this model, we should now have a separate intercept for City, Friends and DUPLO, and a separate slope for only DUPLO (as we had noted earlier that City and Friends had the same slope). This doesn’t change the estimates by much and there is no harm in fitting the model with the extra estimates. But if you are ever interested in cleaning up your model to remove estimates close to 0, this is one option for you.
